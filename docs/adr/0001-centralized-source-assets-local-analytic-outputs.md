# ADR 0001: Centralized Source Assets with Local Analytic Outputs

## Status

Accepted

## Context

The project began with a fully local, co-located data layout in which
RAW files, JPEG companions, XMP sidecars, scripts, and extracted outputs
were all kept near one another for convenience. That layout was useful
for early exploration, but it blurred an important distinction between:

- source-of-truth assets used by the creative workflow
- derived analytic artifacts generated by local validation scripts

As the scripting layer matured, the repository began to model a more
production-like system in which the RAW masters and metadata sidecars
behave like centralized assets, while extracted JSON outputs, manifests,
and validation artifacts remain local and disposable.

## Decision

Source assets should be treated as centrally managed objects, while
downstream analytic outputs should be materialized locally.

The intended target architecture is:

- centralized storage for RAW masters
- centralized storage for XMP sidecars or exported metadata snapshots
- local repository storage for scripts, tests, extracted JSON outputs,
  manifests, and validation artifacts

Within centralized storage, RAW and XMP assets may share one bucket but
should use separate prefixes, for example:

- `s3://<bucket>/raw/...`
- `s3://<bucket>/xmp/...`

## Rationale

- **Source-of-truth separation:** RAW and XMP assets are authoritative
  workflow inputs, not merely local fixtures for scripting convenience.
- **Operational realism:** centralized asset storage better matches
  enterprise or multi-operator production environments than repo-local
  co-location.
- **Storage efficiency:** heavy binary masters should not be duplicated
  unnecessarily inside every local analytic workspace.
- **Localized downstream processing:** extracted JSON, validation
  reports, and manifests are derived artifacts that can be regenerated
  locally without becoming canonical sources themselves.
- **Clearer lineage:** the architecture cleanly distinguishes source
  asset provenance from local analysis products.

## Consequences

### Positive

- source asset provenance becomes clearer
- local analytic outputs remain lightweight and reproducible
- scripts can evolve toward bucket/prefix-aware workflows without
  implying that the repo is the source-of-truth archive
- the design becomes easier to justify in enterprise and technical terms

### Negative

- scripts must manage remote asset access or temporary local syncs
- troubleshooting may require reasoning across local outputs and remote
  asset locations
- early development convenience is reduced compared with pure local
  co-location

## Notes

Temporary local co-location may still be used as a development aid, but
it should be treated as a bootstrap convenience rather than the intended
long-term architecture.
